# Chat Book Cloud (CBC) 前后端重构与架构规范指南

_基于 DDD (领域驱动设计) 思想与次世代 UI 结合的重构演进路线。_

## 一、 重构核心思想：DDD 的轻量级实践与 UI 融合

传统开发模式常导致代码逐渐演变为“大泥球”（Big Ball of Mud），具体表现为：前端页面混合大量脏逻辑、后端 Service 包含超长流水账式代码，跨微服务/模块上下文边界模糊（如 `PROJECT_ISSUES.md` 披露的严重 DO/VO 定义不一致问题）。

本次重构旨在通过 **领域驱动设计边界控制** 结合 **现代化前端模块化治理**，达到高内聚、低耦合，既保障底层稳健，又能灵活支撑上层炫酷的毛玻璃 UI 与 AI 交互应用。

## 二、 项目结构与代码边界划分 (Boundaries)

### 1. 前端代码结构规范 (`chat-book-ui-blog`)
前端不仅仅是展示层，也是用户体验的业务领域层。文件层级必须严格按照责任边界划分：

```text
src/
├── api/          # 【防腐层】 纯粹定义 HTTP 请求调用映射，无 UI 与业务计算。严格对齐后端 Controller 接口
├── assets/       # 静态资源 (图片、字体、全局基础 CSS)
├── components/   # 全局和局部可复用视图组件
│   ├── common/   # 【跨领域通用 UI 组件】，如 毛玻璃按钮、全局气泡、Dialog，体现平台统一 UI 风格
│   └── domain/   # 【领域特定组件】，如 ArticleCard, UserAvatarDetail, AIPanel 等具有鲜明业务目的的复合组件
├── config/       # 【配置层】 抹平硬编码问题，管理环境变量 (ENV)
├── layout/       # 基础页面骨架 (MainLayout, EditorLayout)
├── router/       # 路由策略定义与权限鉴权控制守卫 (路由边界拦截)
├── store/        # 【全局共享状态】 (Pinia) 仅限存贮真正的全局状态，如 UserContext，防止被乱用作数据缓存
├── utils/        # 【纯工具类目录】 详情见第三节规范
└── views/        # 【视图表现层/控制器】 组合不同领域组件，调用 API，接管整个页面级别的应用流操作
```

### 2. 后端基于微服务的领域划分 (如 `cloud-user`, `cloud-article`)
打破基于表操作的 CRUD 惯性，每个服务内部实施清晰的分层：
*   **接入控制层 (Controller)**：薄弱层，职责仅限于路由派发、参数表单验证、以及构造返回封装类 (`CommonResult`)。**绝不包含**任何数据存取和核心业务逻辑。
*   **应用编排层 (Application/Service)**：处理跨实体的宏观业务编排。比如：“发贴” 包含 { 保存内容 -> 增加用户积分 -> 投递给 WebSocket 聊天室 }，此层起协调作用。
*   **领域层 (Domain/Model)**：核心业务规则的体现（如文章审核状态扭转逻辑、判断推荐权限等应当挂载在实体或特定领域服务中，而非全铺在常规 Service 中）。
*   **基础设施层 (Infrastructure)**：处理各类具体的数据库增删改查 (MyBatis Mapper)、远程方法调用 (Feign Client) 以及文件云存储底层交互 (MinIO 客户端)。不向上泄露任何实现细节。

## 三、 工具类归属与存放标准

这是最容易发生混乱的区域。为避免代码发散，做出如下规定：

### 1. 前端工具类存放 (`chat-book-ui-blog/src/utils/`)
*   **准入原则 (纯函数)**：`utils` 目录下的文件（如 `dateFormatter.js`, `regexTool.js`）输入必须决定输出，其内部**绝对禁止**调用 API 接口、**绝对禁止**导入包含生命周期的 UI 库或依赖 Store 以及 Router 上下文。
*   **不符合纯函数怎么办？**：
    *   提取至 `src/composables/` 封装为 Vue3 的组合式函数 (`useAuth.js`, `useFetch.js`)。
    *   如果是仅服务于某个功能的辅助函数，就放在该组件相同的目录下，不提升至全局 `utils`。

### 2. 后端公共类与工具定义
*   所有跨微服务共享的 `VO/DTO/Enum`、工具类和异常类定义，必须放置到全局的公共 API 模块 (`chat-book-cloud-common` 或 `chat-book-cloud-api`) 中。
*   彻底杜绝在不同模块里由开发者手动 Copy 对象导致的同一术语类型及字段数量不对应的序列化灾难风险。

## 四、 编码规范与必须避免的情形 (Red Lines)

以下属于**绝对禁止的触发的高危情况**，开发时需重点审查核对：

### 🚫 防火墙 1：跨层滥用对象结构与数据泄露
*   **❌ 后端绝对禁止**直接将数据库实体 (`XXDO` 或含敏信息的原原本本对象) 作为响应直传给前端。必须存在向 `VO` 的脱敏与属性映射动作。
*   **❌ 前端绝对禁止**组件内消化后端复杂的嵌套多层大对象再去分发，必须在 `views` 或对应的 `store/API` 服务层对后端响应先做一层展平/重配的数据适配，再交给具体的 `common/domain` 卡片展示（屏蔽后端数据结构的异动）。

### 🚫 防火墙 2：硬编码死路灾难
*   **❌ 绝对禁止**前后端代码中的“魔数”与裸露的环境地址。
    *   前端代码如果再出现 `window.location = "http://localhost:8080"` 视为阻断性错误，应使用 `import.meta.env.VITE_APP_BASE_URL`。
    *   后端大量散落的 `if (status == 1)` 必须替换成枚举表达，如 `if (ArticleStatusEnum.PUBLISHED.equals(status))`。

### 🚫 防火墙 3：不当状态管理和数据一致性失控
*   **❌ 绝对禁止**后端但凡涉及到多表或多实体的 `Select` 以外的修改（新增/更新/删除）不上 `@Transactional` 注解，导致操作半途异常系统出现脏数据。
*   **❌ 绝对禁止**抛出异常或错误不加统一拦截。后端所有逻辑错误必须扔出项目统一定义的业务级异常，交由 `GlobalExceptionHandler` 处理为 JSON 信息告知前端。

### 🚫 防火墙 4：UI 层重业务与样式内联
*   **❌ 前端视图禁止**直接在 `.vue` 的 `<template>` 中写极长的且带重要业务条件判断的三目等复杂逻辑语句，应抽取为 `computed` 属性计算。
*   为落实前置制定的沉浸式设计，**禁止**直接通过 `style="margin:xxx"` 随意内联覆盖破坏布局。所有的排版调整请优先使用统一定义的 Tailwind 助手类或主题 CSS 变量（如 `var(--primary-color)`），以保证系统切换深浅色主题无阻滞。
